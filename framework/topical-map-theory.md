# topical-map-theory — pillar, cluster, supporting; raw vs processed

A topical map is the complete, structured inventory of every topic, subtopic, and
semantic relationship in a niche, organised as a content architecture. It defines
*which pages to create, what each is about, and how they link.* It is emphatically
**not** a keyword list or a set of blog-post titles chosen without context.

Read `source-context.md` first — the map is built *from* the central entity and the
core/outer split, not from a keyword tool's export.

## The three tiers

- **Pillar** — a broad page establishing authority over a major branch of the central
  entity. Covers the branch comprehensively at moderate depth; hub of a cluster.
- **Cluster** — a page covering one specific intent/subtopic in real depth; links up to
  its pillar and laterally to siblings where semantically justified.
- **Supporting** — narrow pages (definitions, edge cases, question pages, comparisons)
  that close remaining gaps and feed context/links into clusters and pillars.

This is hub-and-spoke: pillars are hubs, clusters and supporting pages are spokes.
Coverage completeness comes from having a spoke for every meaningful attribute and
query in the branch.

## Core map vs outer map

The whole map is split by the core/outer boundary from the source context:

- **Core map** — pillars and clusters covering the central entity's main attributes,
  densest where monetization lives. Highest priority to build.
- **Outer map** — pillars/clusters/supporting pages on peripheral attributes and
  adjacent topics that build trust and route authority inward. Built after (or thinly
  alongside) the core, especially for solo/80-20 builds.

Every outer node should have a defensible link path *down into* the core — otherwise
it's authority with nowhere to flow (see source-context failure modes).

## Attribute-driven expansion

The map is generated by decomposing the central entity into its attributes (see
`eav-modeling.md`) and turning attributes into topics:

1. List the central entity's attributes (defining / unique / rare / common).
2. Each **defining/unique** attribute → a candidate **pillar** or strong cluster.
3. Each attribute's values, comparisons, and how-tos → **clusters**.
4. Questions, edge cases, and disambiguations around each → **supporting** pages.
5. Related/neighbouring entities → outer-map pillars that link back via shared
   attributes.

Depth is set by niche competitiveness: a contested SaaS or legal niche needs deeper,
more granular spokes than a small local niche. Don't manufacture depth a topic can't
sustain — thin duplicate spokes hurt more than they help.

## Raw map → processed map

Build in two passes:

- **Raw map** — brainstorm *everything*: every attribute, value, question, comparison,
  and query variation you can enumerate (attribute/"token" insertion). Over-generate
  on purpose; completeness first.
- **Processed map** — the executable version: filter to the "right to cover" set,
  dedupe and **merge** near-duplicate nodes (cannibalization prevention at design
  time), assign tier + parent + intent, attach demand signals if grounded, and set
  priority. This is what becomes `topical-map.json`.

Never ship the raw map. Unprocessed maps are how you get 300 overlapping thin pages.

## Preventing cannibalization at the map stage

Two nodes that target the same intent will fight each other after publishing. Catch it
before writing:

- **T0:** compare candidate nodes' target queries + intents by judgement; merge or
  re-scope overlaps (label `asserted`).
- **T1+:** embed each node's query network and flag sibling pairs above the
  `cannibalization_threshold` (config); merge, or differentiate by splitting intent.

One URL, one intent. If two nodes can't be cleanly separated by intent, they're one
page.

## Prioritisation

Score each processed node to order the build:

`priority ≈ business_value × demand_signal × feasibility`

- **business_value** — core > outer; proximity to monetization.
- **demand_signal** — `asserted` at T0; `measured` (autocomplete breadth / trends /
  DataForSEO volume) at T1/T2. Never invent a volume to fill this in.
- **feasibility** — can the brand cover it credibly and beat what's ranking?

Priority seeds `calendar.md`. Publishing order matters for historical-data effects:
establish core pillars early, then fill clusters, then supporting.

## What "done" looks like for a map

- ≥1 pillar per major (defining/unique) attribute of the central entity.
- Every node has: tier, section (core/outer), parent (except top pillars), target
  query, a query network, and an intent (with provenance).
- No sibling pair above the cannibalization threshold.
- Every outer node has a link path into the core.
- A priority order exists and a calendar is seeded.

## Worked micro-example — specialty coffee, "brewing" branch

- Pillar: *Coffee brewing methods* (core).
  - Cluster: *Espresso brewing* → supporting: grind size for espresso, 9-bar
    pressure explained, ristretto vs lungo vs americano.
  - Cluster: *Pour-over brewing* → supporting: V60 vs Chemex, water temperature,
    bloom timing.
  - Cluster: *Cold brew at home* → supporting: ratio, steep time, dilution.
- Outer pillar: *Coffee extraction science* → links down to every brewing cluster
  (shared attribute: extraction).

## Worked micro-example — AP-automation SaaS, "evaluation" branch

- Pillar: *How to evaluate AP automation software* (core, close to money).
  - Cluster: *AP automation ROI / cost justification* → supporting: sample ROI model,
    payback period, hidden costs.
  - Cluster: *AP automation vs manual processing* → supporting: error-rate data,
    touchless-rate benchmarks (grounded only).
  - Cluster: *Integration with [ERP] for AP* → supporting per major ERP.
- Outer pillar: *Accounts payable fundamentals* → links down to evaluation + workflow
  clusters (shared attribute: AP process).
